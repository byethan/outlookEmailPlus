## 2. 已核实根因

### 2.1 主根因

账号管理页当前只做了**客户端分页显示**，没有做**服务端分页查询**：

1. `api_get_accounts()` 一次性返回某分组下的全部账号
2. `loadAccountsByGroup()` 把整组账号一次性放进 `accountsCache`
3. `renderAccountList()` 只是对已加载到内存中的数组做 `slice()`
4. 当分组内账号量上万时，浏览器在 JSON 解析、对象驻留、数组复制和渲染前计算阶段就会明显放大内存与主线程压力，最终页面崩溃

### 2.2 需要同步关注的旧兼容逻辑

追踪导航入口后，当前真实 Dashboard 页面不会走 `loadDashboard()`，而是走：

1. `navigate('dashboard')`
2. `initOverview()`
3. `static/js/features/overview.js`
4. `/api/overview/*`

但 `static/js/main.js` 中仍保留一段旧的 `loadDashboard()` 逻辑，其中包含按分组请求 `/api/accounts?group_id=...` 的实现。当前这段更像**旧兼容代码 / 遗留逻辑**，本轮范围不再把它当成活跃主链路问题，而是作为**顺手清理或下线的旧逻辑**处理。

## 4. 现状分析补充

### 4.1 当前前端不是“真分页”

现状实际上是：

```text
后端一次性查出全量账号
    ↓
前端把整组账号写入 accountsCache[groupId]
    ↓
renderAccountList() 再基于数组长度计算 totalPages
    ↓
只在渲染阶段 slice 当前页
```

因此“每页 50 条”只减少了 DOM 节点数，**没有减少网络体积、浏览器内存占用和 JS 处理量**。

### 4.2 当前缓存结构是后续改造的风险点

当前多处代码都把 `accountsCache[groupId]` 当成“账号数组”直接使用。如果后续服务端分页改造把缓存直接改成：

```javascript
accountsCache[groupId] = {
  accounts: [...],
  pagination: {...}
}
```

那么以下能力会立刻失效或行为异常：

- 标准模式列表刷新
- 紧凑模式列表渲染
- 账号摘要同步回写
- 语言切换后的列表重渲染
- 批量勾选条状态更新

因此后续实现必须先处理缓存结构兼容层，而不是只改一个请求返回值。

### 4.3 并非所有 `load_accounts()` 调用都应分页化

已确认仓库里还有多处全量调用 `load_accounts()`，用于导出、通知分发、外部 API、Token 工具等后台流程。这些调用默认语义仍然是**拿全量账号**，不能因为修复前端崩溃而被误伤。

已确认的典型全量调用方包括：

| 调用方 | 用途 |
|---|---|
| `controllers/accounts.py::api_export_all_accounts()` | 导出全部账号 |
| `controllers/accounts.py::api_export_selected_accounts()` | 导出选中分组账号 |
| `controllers/groups.py::api_export_group()` | 导出单个分组账号 |
| `services/external_api.py` | 外部 API 候选账号读取 |
| `controllers/token_tool.py` | Token 工具账号读取 |
| `services/notification_dispatch.py` | 通知分发账号遍历 |

这意味着后续实现更安全的做法是：

- **新增分页/筛选/排序专用查询能力**
- 或扩展现有仓储函数但**保持默认全量语义不变**

本轮文档先记录该约束，不在文档阶段锁死具体代码写法。

## 5. 已确认实施方向

### 5.1 用户已确认范围

已确认不采用“只压缩字段”或“只做分页不管筛选排序”的收缩方案，后续目标是：

1. **服务端分页**
2. **服务端筛选**
3. **服务端排序**

### 5.2 后续实现至少要覆盖的查询维度

基于当前页面能力，后续账号列表接口至少应支持：

| 维度 | 说明 |
|---|---|
| `group_id` | 分组切换 |

## 7. 风险与约束

| 风险/约束 | 说明 |
|---|---|
| 缓存结构耦合 | 多处代码把 `accountsCache[groupId]` 视为数组 |
| 旧兼容逻辑残留 | 若不顺手清理 `loadDashboard()`，后续容易误判真实调用链 |

| 分页稳定性 | 需要固定稳定排序字段，避免翻页过程数据抖动 |
