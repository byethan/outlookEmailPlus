## 1. 功能定义

### 1.1 核心功能

开发一个 Chrome/Edge 浏览器扩展（Manifest V3），作为 OutlookMail Plus 的独立伴生工具：

1. 用户通过快捷键或点击图标在任意页面唤起极简 Popup 面板
2. 面板内完成「申领邮箱 → 使用 → 获取验证码/链接 → 完成/释放」完整流程
3. 所有操作结果通过一键复制提供给用户，**不做任何页面注入**
4. 申领历史本地持久化，离线可查

### 1.2 本期包含

- ✅ Manifest V3 Popup 扩展骨架（`manifest.json` + `popup.html` + `popup.js`）
- ✅ 通过 `chrome.storage.local` 持久化「配置」与「历史记录」
- ✅ 快捷键唤起 Popup（默认 `Ctrl+Shift+E` / Mac `Command+Shift+E`，通过 `manifest.json` `commands` 字段注册，用户可在 Chrome 扩展快捷键设置中自定义）
- ✅ 申领邮箱（调用 `claim-random`）
- ✅ 获取验证码（调用 `verification-code`，含超时等待逻辑）
- ✅ 获取验证链接（调用 `verification-link`）
- ✅ 完成任务（调用 `claim-complete(result=success)`）
- ✅ 释放邮箱（调用 `claim-release`）
- ✅ 历史记录区（按时间倒序，含邮箱/验证码/链接/状态，支持复制）
- ✅ 主应用侧补充 CORS 支持 `chrome-extension://` 来源
- ✅ 深色/浅色主题切换（`🌙/☀️` 按钮，自动同步主应用 `localStorage['ol_theme']`，CSS 变量对齐主应用设计规范）
- ✅ Popup 宽度 420px（原 380px），CSS 变量全量对齐主应用（`--font-sans`、`--text-secondary` 等）
- ✅ 独立浮窗按钮（`⤢`）：点击后通过 `chrome.windows.create` 弹出可拖拽独立窗口，Popup 内隐藏该按钮（`?mode=window` 参数区分）
- ✅ 申领错误提示优化：优先展示服务器返回的 `message` 字段，而非通用兜底文字
- ✅ API Key 设置面板引导文字（告知从主应用「系统设置 → 对外 API → API 安全」复制）
- ✅ 主应用前置条件补齐：浏览器扩展依赖 `external pool` 已启用；若使用多 Key，还要求该 Key 具备 `pool_access`

### 1.3 本期不包含

- ❌ DOM 自动识别与自动填充（不做）
- ❌ Content Script 注入目标页面（不做）
- ❌ Background Service Worker 长驻（不做）
- ❌ 多服务器实例切换（暂不做）
- ❌ Popup 内部通知推送 / 系统 Notification（不做）
- ❌ 扩展商店发布流程（仅本地加载使用）

## 2. 行为设计

### 2.1 扩展文件结构

```
browser-extension/
├── manifest.json          # V3 清单
├── popup.html             # 正式版 Popup（仅 DOM 结构 + 外部 JS 引用）
├── popup.preview.html     # UI 预览原型（供浏览器直接打开预览，含模拟状态）
├── popup.js               # 主交互逻辑
├── storage.js             # chrome.storage.local 封装（被 popup.js 引用）
├── icons/
│   ├── icon16.png
│   ├── icon48.png
│   └── icon128.png
└── README.md              # 本地加载说明
```

### 2.2 Popup 状态机

Popup 内部维护以下互斥状态：

| 状态 | 触发条件 | 主要 UI |
|------|----------|---------|
| `idle` | 初始或任务结束后 | 「申领邮箱」按钮 + 历史记录折叠区 |
| `claiming` | 已点击申领，等待后端响应 | Loading 动画 |
| `claimed` | 申领成功，展示邮箱地址 | 邮箱地址复制 + 获取验证码/链接 + 完成/释放按钮 |
| `fetching` | 已点击获取验证码/链接，等待后端响应 | Loading 动画（含倒计时）|
| `result_code` | 已拿到验证码 | 验证码复制 + 完成/释放按钮 |
| `result_link` | 已拿到验证链接 | 链接复制 + 在新标签打开 + 完成/释放按钮 |
| `settings` | 点击设置图标 | 配置表单（服务器地址/API Key/默认 Project Key） |

状态持久化策略：打开 Popup 时读取 `chrome.storage.local` 中的 `currentTask`，若存在则恢复至 `claimed` 状态（不自动跳转到其他状态）。

### 2.3 申领邮箱流程

```
用户点击「申领邮箱」
  → 读取 storage: { serverUrl, apiKey, defaultProjectKey }
  → 若 serverUrl 或 apiKey 未配置：弹出提示「请先完成设置」，跳转到设置面板
  → 状态切换到 claiming（loading）
  → POST /api/external/pool/claim-random
      Headers: { X-API-Key: apiKey }
      Body: { caller_id: "browser-extension", task_id: <uuid>, project_key: defaultProjectKey }
  → 成功：写入 currentTask 到 storage，状态切换到 claimed
  → 失败（无可用邮箱 / 网络错误）：显示错误提示，回到 idle
```

`task_id` 使用 `crypto.randomUUID()` 在每次申领时生成。  
`caller_id` 固定为字符串 `"browser-extension"`（v1 硬编码，不在设置面板提供自定义配置项）。

### 2.4 获取验证码流程

```
用户点击「获取验证码」
  → 读取 currentTask.email
  → 状态切换到 fetching（loading + 计时）
  → GET /api/external/verification-code
      Query: { email: currentTask.email, wait: 60 }
      Headers: { X-API-Key: apiKey }
  → 成功（code 不为空）：
      - 将验证码写入 currentTask.code，更新 storage
      - 状态切换到 result_code
  → 超时 / 无结果 / 网络错误：显示错误提示，状态回到 claimed
```

前端超时处理：无论服务端是否支持 `wait` 参数，前端**始终**加 65 s `AbortController` 超时兜底（比服务端 60s 多 5s 缓冲），防止 fetch 无限挂起。

### 2.5 获取验证链接流程

与 2.4 类似，调用 `GET /api/external/verification-link`，成功后状态切换到 `result_link`。  
`result_link` 状态下提供「复制链接」与「在新标签打开（`chrome.tabs.create`）」两个操作入口。

### 2.6 完成 / 释放流程

```
完成（成功）：
  → POST /api/external/pool/claim-complete
      Body: { task_id: currentTask.taskId, result: "success" }
  → 将 currentTask 写入历史（status = "completed"）
  → 清空 currentTask，状态回到 idle

释放（失败/放弃）：
  → POST /api/external/pool/claim-release
      Body: { task_id: currentTask.taskId }
  → 将 currentTask 写入历史（status = "released"）
  → 清空 currentTask，状态回到 idle
```

即使 API 调用失败（网络异常），仍然清空 currentTask 并写入历史（history 里标注 `api_error: true`），避免界面卡死。

### 2.7 设置面板行为

- 打开时读取 `chrome.storage.local` 中已保存的配置回填
- 点击「保存」时：
  1. 调用 `chrome.permissions.request` 申请对该服务器主机的访问权限（`optional_host_permissions` 机制，Chrome 弹出授权对话框）
  2. 用户授权后写入 `storage`，关闭设置面板，回到 `idle` 或 `claimed`（保持原状态）
  3. 若用户拒绝授权，保存操作取消，给出提示「需要授权才能访问该服务器」
- `serverUrl` 末尾斜杠自动 `trim`
- API Key 字段不显示原文（`type=password`），但保存到 storage 时明文存储（`chrome.storage.local` 仅限本地，无 Sync）

## 3. 接口契约

### 3.1 主应用侧 CORS 改动

浏览器扩展的 Popup 发出的请求 `Origin` 为 `chrome-extension://<extension-id>`。  
主应用需在 CORS 中间件中允许该来源通过。

**改动位置**：`outlook_web/app.py` 中 `flask-cors` 配置（或等价 CORS 中间件）

**改动语义**：
- 对 `/api/external/*` 下所有端点，允许 Origin 匹配 `chrome-extension://` 前缀
- 允许 `X-API-Key` 出现在 `Access-Control-Allow-Headers` 中
- 允许 `OPTIONS` 预检（preflight）正常通过
- 不影响现有 Web 界面（`http://` / `https://`）的 CORS 行为

### 3.2 `POST /api/external/pool/claim-random`

| 项 | 值 |
|---|---|
| 请求 Headers | `X-API-Key: <apiKey>`, `Content-Type: application/json` |
| 请求 Body | `{ "caller_id": "browser-extension", "task_id": "<uuid>", "project_key": "<projectKey>（可选）" }` |
| 成功响应 | `200 { "email": "...", ... }` |
| 前置条件 | 主应用已启用 `external pool`；若使用多 Key，则当前 Key 还需具备 `pool_access` |
| 无可用邮箱 | `4xx / 5xx`，前端展示服务端错误信息 |

### 3.3 `GET /api/external/verification-code`

| 项 | 值 |
|---|---|
| 请求 Headers | `X-API-Key: <apiKey>` |
| 请求 Query | `email=<email>&wait=60` |
| 成功响应 | `200 { "code": "123456", ... }` |
| 超时/无结果 | 服务端返回相应状态，前端展示提示 |

### 3.4 `GET /api/external/verification-link`

与 3.3 同结构，响应字段为 `link`。

### 3.5 `POST /api/external/pool/claim-complete`

| 项 | 值 |
|---|---|
| 请求 Body | `{ "task_id": "<taskId>", "result": "success" }` |
| 成功响应 | `200` |

### 3.6 `POST /api/external/pool/claim-release`

| 项 | 值 |
|---|---|
| 请求 Body | `{ "task_id": "<taskId>" }` |
| 成功响应 | `200` |

## 4. 数据语义

### 4.1 `chrome.storage.local` 数据结构

```json
{
  "config": {
    "serverUrl": "http://localhost:5001",
    "apiKey": "sk-xxxx",
    "defaultProjectKey": "my-project"
  },
  "currentTask": {
    "email": "abc@example.com",
    "taskId": "uuid-v4",
    "callerId": "browser-extension",
    "projectKey": "my-project",
    "claimedAt": "2026-04-18T10:00:00.000Z",
    "code": null,
    "link": null
  },
  "history": [
    {
      "id": "uuid-v4",
      "email": "abc@example.com",
      "projectKey": "my-project",
      "claimedAt": "2026-04-18T10:00:00.000Z",
      "completedAt": "2026-04-18T10:05:00.000Z",
      "status": "completed",
      "code": "123456",
      "link": null,
      "apiError": false
    }
  ]
}
```

### 4.2 历史记录管理

- `history` 数组按时间倒序，最新在前
- 最多保留最近 100 条，超出时自动丢弃最旧的
- 字段含义：
  - `status`：`"completed"` / `"released"` / `"error"`
  - `apiError`：若完成/释放时 API 调用失败，置为 `true`，提示可能未同步服务端状态

### 4.3 currentTask 生命周期

| 阶段 | currentTask 状态 |
|------|------------------|
| 申领前（idle） | `null` |
| 申领中 | 存在（`code` / `link` 为 null） |
| 获取到验证码/链接后 | 更新 `code` / `link` 字段 |
| 完成或释放后 | 删除（清空为 null） |

## 5. 兼容与边界

### 5.1 与主应用的解耦

- 扩展不依赖主应用的 Session 鉴权，只使用 `X-API-Key`
- 扩展不读写主应用数据库，只消费现有外部 API
- 主应用 CORS 改动范围严格限定于 `/api/external/*`，不影响内部 Session API

### 5.2 扩展 ID 变更

Chrome 本地加载的扩展每次新安装会生成不同 Extension ID，导致 Origin 变化。  
**CORS 解法**：服务端匹配 `chrome-extension://` 前缀，而非写死具体 ID。

### 5.3 HTTP 与 HTTPS 混合

若主应用使用 HTTP（本地开发），而浏览器默认不限制 `chrome-extension://` → HTTP，可以正常工作。  
若主应用使用 HTTPS，Chrome 不会阻止 `chrome-extension://` → HTTPS 请求，无需额外配置。

### 5.4 popup.html（当前文件）的定位

`browser-extension/popup.html` 目前作为 UI 预览原型存在（含模拟数据和底部切换栏）。  
正式扩展实现时，该文件需要清除预览专用的「状态切换工具栏」，仅保留真实流程逻辑。

## 6. 验收口径（FD 视角）

1. Popup 打开时，若已有未完成任务（currentTask 非 null），自动恢复到 `claimed` 状态并显示对应邮箱
2. 首次使用时，若 config 未配置，点击「申领邮箱」后跳转到设置面板并给出明确提示
3. 申领成功后，邮箱地址可一键复制，且 currentTask 已写入 storage
4. 获取验证码/链接时，若后端超时（wait=60 无结果），前端显示超时提示，状态回到 `claimed`（不清空任务）
5. 完成/释放后，历史记录中出现对应条目（邮箱、时间、验证码/链接、状态均正确）
6. 历史记录按时间倒序展示，最多 100 条
7. 主应用 CORS 配置更新后，从扩展发出的 OPTIONS 预检请求和实际请求均能正常通过
8. `chrome-extension://` CORS 改动不影响现有 Web 界面和 API 的正常访问
9. 验证链接在 `result_link` 状态下「在新标签打开」功能正常（调用 `chrome.tabs.create`）
10. 扩展不注入任何 Content Script，不修改任何目标页面 DOM

## 7. 结论

本 FD 将浏览器扩展定义为：

- **独立子项目**：不依附主应用版本，独立发布与迭代
- **极简 Popup 架构**：仅 Popup + chrome.storage，无 Content Script、无 Background SW
- **后端零新增接口**：100% 复用现有 `/api/external/*` 外部 API
- **主应用最小改动**：仅补充 CORS `chrome-extension://` 来源支持
- **本地优先**：历史记录本地持久化，离线可查，不上报任何使用数据到服务端
